# Plan: Stripe migration/update (Context7 ID `/websites/stripe`)

## Repository findings
1. Project is a single-file Streamlit app (`streamlit_app.py`) with basic Python deps (`streamlit`, `pandas`).
2. No existing Stripe integration was found in code, config, or dependencies.
3. There is no backend webhook endpoint structure yet; current app is data-visualization only.

## Implication for requested migration/update
Because no Stripe code exists, "migration/update" can only be executed in one of these ways:
1. Migrate from a non-Stripe/legacy payment flow that is outside this repo into Stripe.
2. Migrate from an older Stripe approach that exists elsewhere and port/update it into this Streamlit repo.

## Planned execution approach (after approval)
1. Define migration source and target Stripe pattern using Context7 `/websites/stripe` docs.
2. Add Stripe Python SDK dependency and minimal config plumbing (`STRIPE_SECRET_KEY`, optional `STRIPE_WEBHOOK_SECRET`).
3. Implement server-side Stripe session/payment creation path compatible with Streamlit (likely via app-side handler + secure key usage).
4. Add Streamlit UI flow for payment initiation and explicit success/cancel states.
5. If async status is required, implement webhook receiver strategy suitable for deployment model (native endpoint if available, otherwise documented companion endpoint requirement).
6. Add defensive error handling for auth errors, card declines, invalid parameters, and transient API failures.
7. Validate via local run and deterministic manual checks in Stripe test mode.

## Verification checklist
1. App starts with and without Stripe env vars (clear behavior in both cases).
2. Test-mode checkout/payment session is created successfully.
3. Success/cancel UX states render correctly in Streamlit.
4. Failure modes show actionable, non-sensitive errors.
5. Webhook event handling path is validated if in scope.

## Deliverables
1. Updated dependencies and Stripe configuration handling.
2. Stripe payment flow integrated into Streamlit app.
3. Optional webhook handling path or deployment note if webhook endpoint must live outside Streamlit.
4. Brief runbook for test-mode verification.

## Confirmed scope
- Migration source: legacy Stripe code.
- Target flow: Stripe Checkout Sessions.

## Concrete implementation sequence (execution phase)
1. Add `stripe` Python package to `requirements.txt` and keep other dependencies unchanged.
2. Add secure config reads in `streamlit_app.py` using environment variables:
   - `STRIPE_SECRET_KEY` (required to enable checkout)
   - `STRIPE_PRICE_ID` (or equivalent fixed product/price mapping)
   - `APP_BASE_URL` (for `success_url` and `cancel_url`)
3. Initialize Stripe client only when config is present; render a clear UI notice when Stripe is not configured.
4. Add a Streamlit payment panel with a "Pay with Stripe" action that creates a Checkout Session (`mode='payment'`, line items from configured price).
5. Redirect users to `session.url` safely using Streamlit-compatible navigation/link pattern.
6. Detect post-checkout return state via query params (`success`, `canceled`, optionally `session_id`) and render deterministic status messages.
7. If legacy webhook behavior is part of migration, document a companion webhook endpoint requirement (Streamlit app itself is not ideal for robust webhook ingestion).
8. Preserve existing GDP dashboard behavior; Stripe UI should be additive and non-breaking.

## Migration notes to include in final change set
1. Mapping from legacy fields/endpoints to Checkout Session fields.
2. Deprecated legacy behaviors removed or replaced.
3. Environment variable migration table (old -> new).
4. Test-mode validation steps using Stripe test cards.
5. Explicit limitations and production hardening notes (idempotency, webhook verification service, secrets management).
